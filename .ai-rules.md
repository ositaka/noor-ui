- Usually safe to archive after task completion
- But still ask first!

## ü§ñ **For AI Assistants Specifically**

### üö® **CRITICAL: Git Commit Rules**

**NEVER create git commits without explicit user permission.**

**ALWAYS:**
1. Stage the changes with `git add`
2. Show the user what changed with `git status` or `git diff`
3. **ASK permission**: "Should I commit these changes?"
4. **WAIT** for user confirmation
5. Only then run `git commit`

**Example - Good behavior:**
```
AI: "I've made these changes:
- Modified: toast.tsx (RTL positioning)
- Modified: badge/page.tsx (bilingual labels)
- Modified: components/page.tsx (GCC-first ordering)

Should I commit these changes?"
User: "yes"
AI: *runs git commit*
```

**Example - Bad behavior:**
```
AI: *automatically runs git commit without asking*
```

**Why this matters:**
- User may want to review changes first
- User may want to modify the commit message
- User may want to split changes into multiple commits
- User may want to test before committing

**Exception:** ONLY if user explicitly says "commit this" or "make a commit" in their message.

### File Organization Rules

When you encounter a file that seems like it could be deleted:

1. **Check this file first** - Is it protected?
2. **Check the file location** - Is it in `/docs/case-studies/`?
3. **Check the file name** - Does it match a protected pattern?
4. **When in doubt** - Ask the user: "I found [file]. Should I archive, move to case studies, or keep as-is?"
